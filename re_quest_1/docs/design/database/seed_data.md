# seed_data.md

# 初期データ設計

## 1. 概要

本ドキュメントは、`re_quest1` における初期データ方針を定義する。

既存 `quest_1` の seed データを参考にし、
React + ASP.NET Core 版でも同等の動作確認ができる初期データを用意する。

---

# 2. 初期データ方針

第1段階では、既存PoCの確認に使用したデータ構成を維持する。

## 含めるデータ

- 部門
- ロール別ユーザー
- サンプル案件
- サンプルタスク
- サンプル予算
- サンプル通知

---

# 3. 部門データ

## 想定部門

| 部門名                     | 用途                     |
| -------------------------- | ------------------------ |
| プロダクト開発部           | 開発案件・担当者確認用   |
| カスタマーソリューション部 | 部門別アクセス制御確認用 |
| 本部                       | HQ_MANAGER確認用         |

---

# 4. ユーザーデータ

## ロール別ユーザー

| ロール       | 用途                   |
| ------------ | ---------------------- |
| APPLICANT    | 案件申請・自案件確認   |
| DEPT_MANAGER | 部門承認・部門案件確認 |
| HQ_MANAGER   | 本部承認・全社確認     |
| TASK_MEMBER  | タスク更新・工数登録   |

---

# 5. サンプル案件

## 用意する案件

- 下書き案件
- 部門承認待ち案件
- 本部承認待ち案件
- 承認済み案件
- 進行中案件
- 完了案件
- 却下案件

---

# 6. サンプルタスク

## 用意するタスク

- 未着手タスク
- 進行中タスク
- レビュー中タスク
- 完了タスク
- 期限超過タスク

---

# 7. サンプル予算

## 用意するデータ

- 予算内案件
- 予算消化率が高い案件
- CPI低下案件
- 実績工数あり案件
- 経費登録あり案件

---

# 8. サンプル通知

## 用意する通知

| 通知種別          | 用途           |
| ----------------- | -------------- |
| APPROVAL_REQUEST  | 承認依頼確認   |
| APPROVED          | 承認完了確認   |
| REJECTED          | 却下確認       |
| PROJECT_STARTED   | 案件着手確認   |
| PROJECT_COMPLETED | 案件完了確認   |
| TASK_ASSIGNED     | タスク割当確認 |

---

# 9. 動作確認観点

初期データ作成後、以下を確認する。

- ロール別ログインができる
- 申請者が自案件を確認できる
- 部門管理者が自部門案件のみ確認できる
- 本部管理者が全案件を確認できる
- 担当者が自分のタスクを確認できる
- 承認待ち案件が表示される
- 通知一覧が表示される
- ダッシュボード集計が表示される
- アラート判定が表示される

---

# 10. 注意点

- パスワードは開発用の固定値とする
- 本番利用を想定したデータではない
- IDOR対策確認用に、部門外データも用意する
- ロール別アクセス確認ができるようにデータを分散させる
- 既存PoCとの差分が出ないようにする

---

# 11. ASP.NET Coreでの実装方針

初期データは、以下のいずれかの方法で投入する。

## 方法1: Seed処理を作成する

`DataSeeder` クラスを作成し、開発環境で実行する。

例：

```text id="0n73t6"
Data/
└─ DataSeeder.cs
```

---

## 方法2: SQLファイルを作成する

初期投入用SQLを作成する。

例：

```text id="9b4vca"
database/
└─ seed.sql
```

---

## 方法3: EF Core Migrationに含める

少量の固定マスターデータは Migration に含める。

---

# 12. 推奨方針

第1段階では、移植元の `seed.py` に近い形で
`DataSeeder.cs` を作成する方針とする。

理由は以下の通り。

- C#コードで管理できる
- Entityとの整合性を確認しやすい
- ロール別テストデータを作成しやすい
- 移植元seedとの比較がしやすい

---

# 13. 今後の改善候補

移植完了後、以下を検討する。

- 本番用マスターデータとの分離
- テスト用データ生成処理の分離
- 権限制御テスト専用データの追加
- 大量データ確認用seedの追加
- SQLファイルによる投入方式の併用

---

# 14. まとめ

初期データは、React + ASP.NET Core 版で
既存PoCと同等の動作確認を行うために用意する。

第1段階では改善よりも再現を優先し、
既存 `quest_1` の seed データ構成を参考に作成する。
