# ui_policy.md

# UI方針

## 1. 概要

本ドキュメントは、`re_quest1` の React フロントエンドにおけるUI方針を定義する。

第1段階では、既存Vue版のUI構成を維持し、React + TypeScriptで同等の画面・操作性を再現する。

---

# 2. 基本方針

## 第1段階の方針

- 既存Vue版の画面構成を維持する
- 既存Vue版の操作導線を維持する
- UI改善は後回しにする
- 画面遷移を抑えた構成を維持する
- Drawer / Dialog / Popover の役割を維持する
- ロール別表示を維持する

---

# 3. 画面遷移を抑える方針

既存PoCでは、画面遷移を減らし、同一画面内で確認・判断・操作ができるUIを重視した。

React版でもこの方針を維持する。

## 目的

- 操作の中断を減らす
- 一覧確認から詳細確認までの流れを維持する
- 承認・確認・判断を同一画面内で行いやすくする
- 業務利用時の画面移動負担を減らす

---

# 4. Drawer利用方針

Drawerは、現在の画面を維持したまま補助的な入力・操作を行うために利用する。

## 主な用途

| Drawer              | 用途             |
| ------------------- | ---------------- |
| ProjectCreateDrawer | 案件申請         |
| NotificationDrawer  | モバイル通知一覧 |

---

## 案件申請Drawer

案件申請は専用ページへ遷移せず、DashboardPage上でDrawer表示する。

```text
/?action=create-project
```

## 方針

- ダッシュボード表示を維持したまま申請できる
- URLから直接Drawer表示できる
- 登録完了後はダッシュボードを再取得する

---

# 5. Dialog利用方針

Dialogは、一覧画面を維持したまま詳細確認を行うために利用する。

## 主な用途

| Dialog                   | 用途       |
| ------------------------ | ---------- |
| NotificationDetailDialog | 通知詳細   |
| TaskDetailDialog         | タスク詳細 |
| ConfirmDialog            | 確認操作   |

---

## 方針

- 一覧画面の状態を維持する
- 詳細確認後に元の一覧へ戻りやすくする
- 操作確認を明確にする

---

# 6. Popover利用方針

Popoverは、PC表示において軽量な一覧確認に利用する。

## 主な用途

| Popover             | 用途       |
| ------------------- | ---------- |
| NotificationPopover | PC通知一覧 |

---

## 方針

- サイドバーから通知一覧を確認する
- 通知一覧ページへの遷移を避ける
- 詳細はDialogで表示する

---

# 7. ダッシュボードUI方針

DashboardPageは、ロール別の業務開始地点として扱う。

## 方針

- ログイン後の初期画面とする
- KPIカードを表示する
- ロール別に表示内容を切り替える
- 案件一覧やタスク一覧へ自然に遷移できる構成にする

---

# 8. KPIカード方針

KPIカードは、単なる数値表示ではなく一覧表示切替の起点として利用する。

## 方針

- KPIカード選択時に一覧表示を切り替える
- 数値確認から対象データ確認へつなげる
- 表示中フィルタを明確にする

---

# 9. 案件一覧UI方針

案件一覧は、DashboardPage内に組み込んで表示する。

```text
/?view=projects
```

## 方針

- 案件一覧専用ページへの遷移を減らす
- ダッシュボード上で案件確認できる
- 案件詳細は `/projects/:id` へ遷移する

---

# 10. 通知UI方針

通知UIは、PCとモバイルで表示方法を分ける。

## PC

- SidebarからNotificationPopoverを表示
- 通知一覧をPopoverで確認
- 通知詳細はDialogで表示

---

## モバイル

- HeaderからNotificationDrawerを表示
- 通知一覧をDrawerで確認
- 通知詳細はDialogで表示

---

## 共通

- 未読件数を表示する
- 既読処理を行う
- 全件既読を行う
- WebSocket通知を反映する

---

# 11. ガントチャートUI方針

ガントチャートは、案件詳細画面内で表示する。

## 方針

- 案件詳細内でタスク進捗を確認できる
- タスククリック時にTaskDetailDialogを表示する
- 第1段階では高度なドラッグ操作は行わない

---

# 12. ロール別UI方針

ロールごとに必要な情報のみ表示する。

## APPLICANT

- 自分の申請案件
- 承認状況
- 案件申請

---

## DEPT_MANAGER

- 自部門案件
- 部門承認待ち
- リスク案件

---

## HQ_MANAGER

- 全社案件
- 本部承認待ち
- 部門別集計
- 全社リスク案件

---

## TASK_MEMBER

- 自分のタスク
- 本日のタスク
- 担当案件

---

# 13. エラー表示方針

## 方針

- APIエラーは分かりやすく表示する
- 401はログイン画面へ遷移する
- 403は権限不足として表示する
- 404は対象データなしとして表示する
- バリデーションエラーはフォーム上に表示する

---

# 14. Loading表示方針

## 方針

- API取得中はLoadingを表示する
- 画面全体Loadingと部分Loadingを使い分ける
- 長時間の空白表示を避ける

---

# 15. Empty表示方針

データが存在しない場合は、空表示用コンポーネントを利用する。

## 例

```text
対象データがありません
通知はありません
タスクはありません
```

---

# 16. 第1段階方針

第1段階では既存Vue版のUI構成を維持する。

変更しない内容

- Drawer利用
- Dialog利用
- Popover利用
- KPIカード
- 案件一覧表示
- 通知UI
- ガントチャート表示
- ロール別表示

---

# 17. 今後の改善候補

移植完了後に検討する。

- UIデザイン刷新
- レスポンシブ対応強化
- アクセシビリティ改善
- 表示密度の最適化
- ガントチャート操作性改善
- 通知UI改善
- 案件一覧の検索性向上
- テーマカラー整理

---

# 18. まとめ

React版では、既存Vue版で実装した画面遷移を抑えたUI方針を維持する。

第1段階ではUI改善よりも既存PoCの再現を優先し、
Drawer / Dialog / Popover を活用した既存導線を再実装する。
