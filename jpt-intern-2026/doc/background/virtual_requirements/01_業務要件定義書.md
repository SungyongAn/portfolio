# 開発管理統合アプリケーション PoC — 業務要件定義書

## 1. 企業プロフィール

### 会社概要

| 項目 | 内容 |
|:-----|:-----|
| 社名 | **株式会社ネクストフロー**（架空） |
| 業態 | 自社SaaSプロダクト開発（BtoB向け業務システム） |
| 従業員数 | 約120名 |
| 組織構造 | 開発本部の下に3つの開発部門 |
| 主力プロダクト | クラウド型ワークフローシステム「FlowBase」 |

### 組織図（PoC対象範囲）

```
開発本部（本部管理者: 1名）
├── プロダクト開発部（部門管理者1名 + メンバー15名）
│   └── FlowBaseの新機能開発・改善
├── プラットフォーム部（部門管理者1名 + メンバー10名）
│   └── インフラ・共通基盤・CI/CD
└── カスタマーソリューション部（部門管理者1名 + メンバー8名）
    └── 顧客固有のカスタマイズ開発・導入支援
```

## 2. ペルソナ定義

### ペルソナ1: 申請者（開発リード）

| 項目 | 内容 |
|:-----|:-----|
| **名前** | 田中 翔太（30歳） |
| **所属** | プロダクト開発部 |
| **役職** | シニアエンジニア / テックリード |
| **業務概要** | FlowBaseの機能開発をリードし、新規案件の企画・提案から実装まで担う |
| **日常業務** | コードレビュー、設計、スプリント管理、顧客要望の技術調査 |
| **このシステムでの行動** | 新機能やリファクタリング案件を起案し申請。承認後は進捗とコストを管理 |

**ミッション・目的**:

> 「技術的に正しいものを、顧客に届く形で、期限内に出荷する」

- **最優先**: 担当プロダクトの機能をリリースし、顧客の課題を解決すること
- **技術品質の担保**: 短期的な開発速度と中長期的な保守性のバランスを取る。技術的負債が蓄積すると将来のリリース速度が落ちることを経験的に知っている
- **チームのブロッカー除去**: 自分が起案した案件が承認待ちで止まると、チームメンバーのアサインが宙に浮く。承認プロセスの透明性と速度が自分のチーム運営に直結する
- **管理工数の最小化**: 報告のための報告書作成（Excel→再投稿）に時間を取られるのは、本来の開発時間を食っている。書いたことが自動的に上に伝わる仕組みがほしい

**このシステムに求めること**:
- 申請したら、あとは承認ステータスが自動で追える（催促メールを自分で送りたくない）
- 進捗を1箇所に書けば、部門管理者にも本部にも自動で伝わる（二重入力の廃止）
- 予算の消化状況を自分でもリアルタイムに確認し、超過前にアラートがほしい

**ペインポイント（現行業務の不満）**:
- 案件を申請システムで出した後、承認状況が不透明。「今どこで止まってるの？」が分からない
- 承認後の進捗報告をExcelに書き、さらに申請システムにも再投稿するのが無駄
- 外出先やリモートワーク時にスマホ・タブレットから進捗を更新したい

### ペルソナ2: 申請者（ジュニア）

| 項目 | 内容 |
|:-----|:-----|
| **名前** | 佐藤 美咲（25歳） |
| **所属** | カスタマーソリューション部 |
| **役職** | エンジニア（2年目） |
| **業務概要** | 顧客向けカスタマイズ案件の実装を担当。小規模な案件は自分で起案もする |
| **日常業務** | 実装、テスト、顧客MTGでの技術説明 |
| **このシステムでの行動** | 小〜中規模の案件を起案。承認されたら自分のタスクの進捗を更新 |

**ミッション・目的**:

> 「顧客と約束した納期を守り、信頼されるエンジニアになる」

- **最優先**: 自分が任された実装を品質を保って期限内に完了すること
- **顧客との直接的な信頼関係**: カスタマーソリューション部は顧客と直接対面する。実装の遅延は即座に顧客満足度に影響し、担当営業からもプレッシャーがかかる
- **成長と自走**: 2年目として、小さい案件は自分で起案から完了まで回せるようになりたい。ただし起案の「お作法」（何を書けばいいか、どう見積もればいいか）がまだ手探り
- **自分の仕事の全体像の理解**: 自分が担当している案件が部門全体・会社全体のどこに位置づけられているのかを知りたい。優先度が見えないと、複数案件を抱えたときの判断ができない

**このシステムに求めること**:
- 入力が簡単で迷わないUI（何が必須で何が任意か明確）
- 自分の案件の承認状況が一目でわかる安心感
- 進捗入力が日常の作業フローの中で自然にできること（わざわざExcelを開く別作業にしたくない）

**ペインポイント**:
- Excelの進捗管理フォーマットが部門ごとに違い、異動時に混乱した
- 入力項目が多すぎて何を書けばいいか分からない。もっと簡単にしてほしい
- 自分の案件が全体の中でどういう優先度なのかが見えない

### ペルソナ3: 部門管理者

| 項目 | 内容 |
|:-----|:-----|
| **名前** | 鈴木 健一（42歳） |
| **所属** | プロダクト開発部 部長 |
| **役職** | 部門管理者 |
| **業務概要** | 部門の案件ポートフォリオ管理、リソース配分、メンバーの進捗把握、本部への報告 |
| **日常業務** | 1on1、案件の優先度調整、予算管理、部門定例 |
| **このシステムでの行動** | 部下の申請を確認・一次承認。部門全体の進捗と予算消化を監視 |

**ミッション・目的**:

> 「限られたリソースで最大の成果を出し、本部に対する部門の信頼を維持する」

- **最優先**: 部門に配分された年間予算と人員で、コミットした案件を計画通りに完了させること
- **投資判断（一次スクリーニング）**: 部下から上がってくる案件が「今この部門のリソースを投下する価値があるか」を判断するゲートキーパー。技術的妥当性だけでなく、**部門のキャパシティに対して現実的か**を見る。過剰な並行案件を承認すると全てが中途半端になる
- **予算執行責任**: 部門予算の執行状況を本部に説明する責任がある。年度末に「予算が余った」は機会損失、「超過した」は管理能力を疑われる。**計画対比での執行率**を常に把握していなければならない
- **リソース最適配分**: 15名のエンジニアを複数案件にどう配分するか。遅延している案件にリソースを追加投入するか、別案件の優先度を下げるか、という判断を日々行う
- **本部への説明責任**: 「部門で何が起きているか」を問われたとき、数字とファクトで即答できなければならない。現状はExcel集約に時間がかかり、リアルタイム性がない

**このシステムに求めること**:
- 部門全体の案件・進捗・予算を**1画面で俯瞰**できるビュー
- 遅延案件やリスクのある案件が**自動的に目立つ**仕組み（アラート・色分け等）
- 本部報告のためのデータ集約が不要になること（システムが自動で集約済み）
- 承認依頼を見落とさない通知

**ペインポイント（現行業務の不満）**:
- メンバーのExcelを集めて本部用に統合するのに**毎週2時間**かかる
- Excelを本部にメールで送る運用が面倒。送り忘れると催促される
- 部門内メンバーのタスク状況を**一目で把握したい**が、各人のExcelを開かないと分からない
- 承認依頼が来ても気づかないことがある（メールに埋もれる）

### ペルソナ4: 本部管理者

| 項目 | 内容 |
|:-----|:-----|
| **名前** | 高橋 裕子（48歳） |
| **所属** | 開発本部 本部長 |
| **役職** | 本部管理者 |
| **業務概要** | 3部門横断での案件優先度決定、全体予算管理、経営層への報告 |
| **日常業務** | 部門長会議、経営会議への報告資料作成、案件の最終承認判断 |
| **このシステムでの行動** | 部門承認済み案件の最終承認。全部門の進捗・予算をダッシュボードで監視 |

**ミッション・目的**:

> 「開発投資のROIを最大化し、経営層に対して開発本部の成果を数字で証明する」

- **最優先**: 開発本部全体の年間予算（人件費含む数億円規模）を、事業成長に最も貢献する案件に配分し、その投資対効果を経営層に示すこと
- **投資判断（最終決裁）**: 部門管理者を通過した案件に対し、**全社戦略との整合性**と**部門横断での優先度**を判断する。「プロダクト開発部のこの案件と、カスタマーソリューション部のこの案件、今期どちらに投資すべきか？」という問いに答える立場
- **ポートフォリオ管理**: 全案件を「新規機能開発」「技術的負債の解消」「顧客対応」「インフラ改善」等のカテゴリで俯瞰し、投資配分が偏っていないかを監視する。新機能ばかりに投資して基盤が崩れる、あるいは保守ばかりで成長投資がない、といったバランス崩壊を防ぐ
- **予算執行のガバナンス**: 各部門の予算消化ペースを四半期単位で追跡し、年度末に予算超過/大幅余剰が発生しないよう先手を打つ。超過が見込まれる場合は早期に経営層と調整し、追加予算の確保または案件の延期判断を行う
- **経営会議への報告義務**: 月次の経営会議で「開発本部は今期何にいくら使い、何が完了し、何が遅延しているか」を報告する。この資料作成に現状は半日かかっており、しかもデータの鮮度が低い（先月時点のExcelベース）
- **リスクの早期検知**: 予算超過・大幅遅延・炎上案件は、放置すると経営課題に発展する。「知らなかった」は許されない立場。問題を最速で検知し、対処の指示を出せる状態を常に保ちたい

**このシステムに求めること**:
- **ダッシュボード**: 全部門の案件進捗・予算消化を一画面で把握。「今、開発本部で何が起きているか」が5秒で分かる状態
- **予算消化のリアルタイム可視化**: 部門別・案件別の消費率が常に最新。閾値超過時のアラート
- **判断材料の即時提供**: 案件の承認判断時に、その案件が全体予算に与えるインパクトが分かる
- **報告工数の削減**: 経営会議向けの数字がシステムから自動で取れる状態（理想はダッシュボードをそのまま見せられるレベル）

**ペインポイント（現行業務の不満）**:
- 各部門からバラバラのExcelが届き、**全体像を把握するのに半日かかる**
- 予算消化率がリアルタイムで見えず、四半期末に初めて「超過していた」と気づくことがある
- 経営会議向けの報告資料を毎回手作業でExcelからPowerPointに転記している
- 「あの案件、今どうなっている？」と聞かれたとき即答できない

## 3. 現行業務の課題

| # | 課題 | 影響 |
|:--|:-----|:----|
| C1 | 案件申請は専用の申請システム、進捗は部門別Excel、予算は別Excelと**3系統に分断** | データの横断的把握が不可能 |
| C2 | 各部門のExcelフォーマットが**不統一** | 本部でのデータ集約に手作業が必要 |
| C3 | 進捗状況を元の申請システムに**再投稿**（二重入力） | 本部報告のための無駄な工数 |
| C4 | 予算消費率がExcel管理のため**リアルタイム追跡が困難** | 予算超過リスクの早期検知ができない |

## 4. 業務フロー定義

### 4-1. 案件ライフサイクル

```
[起案] → [申請] → [部門承認] → [本部承認] → [開発中] → [完了]
                    ↓ 却下          ↓ 却下
                 [却下・再起案]   [却下・再起案]
```

### 4-2. フェーズ別の業務詳細

#### Phase 1: 起案・申請（申請者）

| # | 業務 | 詳細 | インプット | アウトプット |
|:--|:-----|:-----|:---------|:----------|
| 1-1 | 案件の企画 | 顧客要望・技術課題・戦略目標から案件を起案 | 顧客フィードバック、技術調査結果 | 案件概要 |
| 1-2 | 概算見積り | 工数（人月）と金額を概算で見積もる | 過去の類似案件実績 | 概算予算 |
| 1-3 | 申請情報の入力 | システムに案件名・目的・概算予算・部門を入力し申請 | 案件概要 + 概算予算 | 申請レコード（ステータス: 申請中） |

| 項目 | 型 | 必須 | 補足 |

| 案件名 | テキスト | ○ | 50字以内 |

| 概算予算（金額） | 数値（円） | ○ | |
| 概算工数（人月） | 数値 | △ | 任意だが入力推奨 |
| 開発開始予定日 | 日付 | △ | |
| 完了予定日 | 日付 | △ | |

#### Phase 2: 承認（部門管理者 → 本部管理者）

| # | 業務 | 担当 | 判断基準 | アクション |
|:--|:-----|:-----|:---------|:----------|
| 2-1 | 一次承認 | 部門管理者 | 部門の開発リソースに余裕があるか、技術的妥当性、優先度 | 承認 or 却下（却下理由を記入） |
| 2-2 | 最終承認 | 本部管理者 | 全社の予算枠に収まるか、部門横断での優先度、事業戦略との整合性 | 承認 or 却下（却下理由を記入） |

**承認時の操作**:
- 案件の申請内容を閲覧
- 「承認」または「却下」ボタンを押す
- 却下の場合はコメント（理由）を必須入力
- 承認時のコメントは任意

**却下後のフロー**:
- 申請者に通知が届く（却下理由付き）
- 元の申請は「却下」ステータスで保持（履歴として残る）
- 申請者は内容を修正して**新規案件として再申請**する（却下案件の編集・再提出ではない）

#### Phase 3: 開発管理（承認後）

##### 3-a. 進捗管理

**管理方式: タスクリスト + 進捗率バー（ガントチャートではない）**

このシステムの進捗管理の目的は**スケジューリング（いつ誰が何をするかの計画立案）ではなく、進捗の報告・集約・可視化**である。申請者が進捗を1箇所に入力すれば、部門管理者・本部管理者の画面にリアルタイムで反映される仕組みとする。

ガントチャートを採用しない理由:
- タスク間の依存関係管理はこのPoCの目的外
- 申請者の役割は「自分のタスクの進捗を報告する」ことであり、スケジュールを引くことではない
- 管理者が求めているのは「案件一覧の横並び比較」と「遅延の即時検知」であり、個別タスクのタイムライン表示ではない

| # | 業務 | 担当 | 詳細 |
|:--|:-----|:-----|:-----|
| 3a-1 | タスク登録 | 申請者 | 承認された案件をタスクに分解して登録 |
| 3a-2 | 進捗更新 | 申請者 | 各タスクのステータス・進捗率を定期的に更新 |
| 3a-3 | 遅延検知 | システム | 期限超過タスクを視覚的に強調表示（赤色等） |
| 3a-4 | 部門内確認 | 部門管理者 | 自部門の全案件を進捗率バー付き一覧で俯瞰。遅延案件を即座に検知 |
| 3a-5 | 全社確認 | 本部管理者 | 全部門横断で進捗率バー付き一覧を確認。経営報告に活用 |

**タスクの管理項目**:

| 項目 | 型 | 必須 | 補足 |
|:-----|:----|:-----|:-----|
| タスク名 | テキスト | ○ | |
| 担当者 | ユーザー選択 | ○ | 同一部門のメンバーから選択 |
| ステータス | 選択 | ○ | 未着手 / 進行中 / レビュー中 / 完了 |
| 進捗率 | 数値（0〜100%） | ○ | |
| 開始日 | 日付 | ○ | |
| 期限 | 日付 | ○ | |
| 備考 | テキスト | — | |

**案件全体の進捗率**: 配下タスクの進捗率の加重平均（またはタスク完了率）で自動算出し、案件一覧画面では進捗率バーとして表示する。

##### 3-b. 予算（工数・金額）管理

**自社開発企業の予算構造**:

自社開発企業のプロジェクトコストの**80〜90%は人件費（工数）**である。したがって、工数管理は予算管理そのものである。

```
案件の総コスト = 工数コスト（人月 × 単価） + 直接経費（外注費・ライセンス費・機材費等）
```

参考: ネクストフロー社の場合
- エンジニア1人あたりの年間コスト: 約1,000〜1,200万円（給与+社保+間接費）
- 人月単価（社内計算用）: 約80〜100万円/人月
- 開発本部の年間予算（人件費+直接経費）: 約4.5〜5億円

**データの流れ（実績は現場から上がる）**:

```
申請者（現場）               部門管理者              本部管理者
  │                          │                      │
  ├─ 工数実績を入力 ──────→ 部門集計を確認 ──────→ 全社集計を確認
  │  （月次 or 随時）         │                      │
  ├─ 直接経費を入力 ──────→ 部門予算消化率 ──────→ 全社予算消化率
  │  （発生都度）             を監視                  を監視・経営報告
```

| # | 業務 | 担当 | 詳細 |
|:--|:-----|:-----|:-----|
| 3b-1 | 工数実績の入力 | 申請者 | 案件に対する工数実績（人月）を月次で入力。自分および自分が管理するメンバーの工数 |
| 3b-2 | 直接経費の入力 | 申請者 | 外注費・ライセンス費等の直接経費を発生都度入力 |
| 3b-3 | 案件別予実確認 | 申請者 | 自分の案件の予算消化状況（工数・金額）を確認し、超過リスクを自己検知 |
| 3b-4 | 部門予実確認 | 部門管理者 | 自部門の全案件の工数・金額の予実を一覧で確認。超過リスク案件を早期発見 |
| 3b-5 | 全社予実確認 | 本部管理者 | 全部門横断で予算執行状況を確認。経営報告に活用 |
| 3b-6 | 予算超過アラート | システム | 消費率が閾値（例: 80%）を超えた案件を視覚的にアラート |

**予算の入力項目**:

*申請時（計画）*:

| 項目 | 型 | 必須 | 補足 |
|:-----|:----|:-----|:-----|
| 予算工数（人月） | 数値 | ○ | 案件全体の計画工数 |
| 予算金額（万円） | 数値 | ○ | 工数コスト + 直接経費の合計 |

*実績入力（月次 — 申請者が入力）*:

| 項目 | 型 | 必須 | 補足 |
|:-----|:----|:-----|:-----|
| 対象月 | 年月 | ○ | 例: 2026年4月 |
| 実績工数（人月） | 数値 | ○ | その月に投入した工数 |
| 直接経費（万円） | 数値 | — | 外注費・ライセンス費等。発生した場合のみ |
| 備考 | テキスト | — | 超過理由・見込み変更等 |

**案件の予算サマリ（自動算出）**:

| 指標 | 算出方法 |
|:-----|:--------|
| 予算工数 | 申請時の計画工数 |
| 実績工数 | 月次実績の累計 |
| 工数消化率 | 実績工数 / 予算工数 x 100% |
| 予算金額 | 申請時の計画金額 |
| 実績金額 | 実績工数 x 人月単価 + 直接経費の累計 |
| 金額消化率 | 実績金額 / 予算金額 x 100% |
| 残工数 | 予算工数 - 実績工数 |
| 残予算 | 予算金額 - 実績金額 |

## 6. 業務ルール

| # | ルール | 詳細 |
|:--|:-------|:-----|
| BR-1 | 承認は一方向 | 部門管理者→本部管理者の順。逆戻しなし |
| BR-2 | 却下後は新規再申請 | 却下案件の編集・再提出はできない。新しい案件として再申請する |
| BR-3 | 進捗管理は承認後のみ | 本部管理者が最終承認した案件のみ、進捗入力が可能になる |
| BR-4 | 予算実績は承認後のみ | 同上。承認前の案件には実績登録できない |
| BR-5 | 消費率の自動算出 | 実績額 / 予算額 x 100%。入力のたびに再計算 |
| BR-6 | 案件進捗率の自動算出 | 配下タスクの進捗率から自動計算 |
| BR-7 | データアクセス制御 | 申請者は自身の案件のみ、部門管理者は自部門のみ、本部管理者は全部門 |
| BR-8 | 代理承認なし | PoC範囲では不要 |
| BR-9 | 却下理由は必須 | 却下時にはコメント入力を必須とする |
