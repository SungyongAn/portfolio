## 各画面の入力・表示項目

### SCR-01：ログイン
**入力**
- メールアドレス
- パスワード

**表示**
- エラーメッセージ（認証失敗時）

**操作**
- 「パスワードをお忘れの方はこちら」リンク

**遷移先**

- パスワードリセット申請（SCR-12）

### SCR-02：ダッシュボード（生徒・図書委員）

- 代理操作メニュー（`is_committee: true`の場合のみ表示）

**遷移先**
- 資料検索（SCR-05）
- 貸出処理（SCR-16）
- 返却処理（SCR-17）
- マイページ（SCR-07）

- パスワード変更（SCR-08）

**遷移先**

### SCR-06：資料詳細
**対象ロール**：全員

**表示**
- タイトル・著者・ISBN・NDC・出版社・出版年
- 所蔵校・貸出状況
- 現在の予約・順番待ち人数
- 返却予定日（貸出中の場合）

**操作（student / committee / librarian）**
- 「予約する」ボタン（自校蔵書・利用可能時）
- 「順番待ちに登録」ボタン（貸出中時）
- 「他校から取り寄せる」ボタン（他校資料・締め切り前）
- 「次週受付」メッセージ（他校資料・金曜15:00以降）

**操作制限（フロント表示・バックエンドでも検証）**
- 貸出中5冊に達している場合 → ボタンをグレーアウトし「貸出上限（5冊）に達しています」を表示
- 延滞中の資料がある場合 → ボタンをグレーアウトし「延滞中の資料があります。返却後にご利用ください」を表示

**操作（代理操作UI・committee / librarian のみ表示）**
- 対象生徒検索・選択欄（自校生徒のみ）
- 選択生徒の貸出冊数・延滞状況を表示
- 選択後、上記ボタンが代理操作として実行される（上限・延滞制限は代理操作時も適用）

> **セキュリティ注意点**：代理操作UIはフロントエンドの`is_committee` / `role`で表示制御するが、バックエンドAPIでも必ず権限・対象生徒の所属校を検証する

### SCR-14：貸出管理（図書委員）

**制限**

---

### SCR-15：貸出管理（司書）
**対象ロール**：librarian

**表示**
- 自校生徒の貸出・予約状況一覧（SCR-14と同様）
- 延滞中資料一覧（返却期限超過の一覧）
- 貸出状況サマリ（貸出中総数・延滞件数）

**操作**
- 生徒を選択 → 選択生徒の詳細を展開
- 「この生徒の代理で資料を検索する」→ SCR-05へ遷移
- 全件CSV出力ボタン

**制限**
- 閲覧・操作できるのは自校生徒のみ

---

### SCR-16：貸出処理
**対象ロール**：student / committee / librarian

**入力**
- 資料バーコードをスキャン（またはID手入力）

**表示**
- 資料情報（タイトル・著者・所蔵校）
- 貸出可否（上限・延滞・他の予約者有無を確認後に表示）
- 返却期限（貸出確定後・貸出日から2週間）

**操作（committee / librarian のみ表示）**
- 対象生徒検索・選択欄（自校生徒のみ）
- 選択生徒の貸出冊数・延滞状況を表示
- 貸出上限・延滞制限は代理操作時も同様に適用

**操作制限（フロント表示・バックエンドでも検証）**
- 貸出上限（5冊）超過時 → 「貸出上限に達しています」を表示し貸出不可
- 延滞中資料あり → 「延滞中の資料があります」を表示し貸出不可
- 他の予約者あり → 「他に予約者がいるため貸出できません」を表示し貸出不可

**遷移先**
- 貸出確定後 → 同画面リセット（連続処理対応）

---

### SCR-17：返却処理
**対象ロール**：student / committee / librarian

**入力**
- 資料バーコードをスキャン（またはID手入力）

**表示**
- 資料情報（タイトル・著者・所蔵校）
- 借りた生徒名・返却期限・延滞有無
- 返却確定後：次の予約者がいる場合「○○さんが予約中です。専用棚へ移動してください」を表示

**操作（committee / librarian のみ表示）**
- 代理返却である旨の確認メッセージを表示

**遷移先**
- 返却確定後 → 同画面リセット（連続処理対応）

## 貸出ルール

| ルール | 内容 |
|--------|------|
| 貸出上限 | 1人あたり同時5冊まで（図書館間貸出を含む） |
| 延滞制限 | 延滞中（返却期限超過）の資料がある場合、新規貸出・予約不可 |
| 制限の適用 | 代理操作時も同様に適用。フロント表示制御＋バックエンドAPI検証の両方で担保 |
