# 課題2 提案内容

## 1. 機能改善提案

### 1.1 重複登録防止
**課題：**
複数のマネージャーが同じ部員・同じ計測日の測定記録を同時に入力した場合、重複登録が発生する可能性がある。

**提案：**
- `create_measurement()` に同一部員・同一計測日の重複チェックを追加
- `rejected` ステータスのレコードが存在する場合は上書き更新して `draft` に戻す
- `draft` / `pending_member` / `pending_coach` / `approved` の場合は400エラーを返す
- 確認モーダルを追加（登録前に入力内容を確認してから承認依頼を送信）

**実装箇所：**
- `backend/app/services/measurement_service.py`
- `frontend/src/views/manager/MeasurementResultSubmit.vue`

**実装状況：実装済み**

### 1.2 チーム全体の傾向可視化ダッシュボード

現状、測定記録はテーブル形式での閲覧のみで、チーム全体の傾向把握が困難。
また部員自身も自分のチーム内での位置づけを把握する手段がない。

**提案：**
EChartsを使った可視化ダッシュボードを各ロール向けに追加する。

**具体的な可視化内容：**

【監督・コーチ向け】

- 部員個人の測定結果時系列推移（折れ線グラフ）
- 部員同士の能力比較（レーダーチャート）
- チーム内順位表（テーブル）

【部員向け】
- 測定項目別のチーム内順位表示
- チーム平均との比較表示
- 自身の測定結果時系列推移（折れ線グラフ）

**将来的な拡張：**
- ポジション別フィルター・比較機能を追加（1.9参照）

**実装箇所：**
- `frontend/src/views/shared/DashboardView.vue`（グラフ追加）
- `frontend/src/components/MeasurementChart.vue`（新規作成）

**実装状況：実装中**

## 2. 運用ルール整備提案

### 2.1 測定記録の入力ルール
- 測定記録の入力は測定日当日または翌日までに行う
- 入力後は速やかに承認フローを発行する
- 入力ミスが発覚した場合はマネージャーが作り直す（否認→再作成）

### 2.2 承認フローの運用ルール
- 部員は承認依頼を受けたら**2営業日以内**に承認・否認を行う
- コーチは部員承認後**2営業日以内**に最終承認・否認を行う
- 否認の場合はマネージャーに理由を口頭で伝える

### 2.3 アカウント管理ルール
- マネージャーは個別アカウントを使用する（共有アカウント禁止）
- パスワードは定期的に変更する
- 卒業・退部時はコーチ・監督がアカウントのステータスを更新する

### 2.4 測定日の運用ルール
- 測定は2か月に1度、全部員参加で実施する
- 欠席者の測定は別日に実施し、同じ計測日で登録する
- 同一部員・同一計測日の重複登録は禁止する
