# ログインレスポンス設計

## 概要
ログイン時に返却するユーザー情報の設計方針を定義する。ユーザーロール（生徒/教師/管理者）ごとに適切な情報構造を提供し、フロントエンドでの条件分岐を最小化することを目指す。

## ロール別の情報取得方針

### 生徒ログイン時
- 現在所属している学年・クラス情報を取得
- シンプルな所属情報のみを返却

### 教師ログイン時
- `teacher_assignments` を基に担当学年・クラス情報を取得
- 複数の担当（担任・学年主任・教科担当）を考慮した設計

### 管理者ログイン時
- 管理者権限に基づく情報を返却
- 担当情報は不要

## teacher_assignments のデータ構造設計

### 基本方針
- `teacher_assignments` は**常に list として扱う**
  - 未割当時も空配列 `[]` を返却する
  - `list | None` は使用しない
- バックエンドで値が取得できない場合でも、レスポンス構造を固定化する
- フロントエンド側での条件分岐を最小化する

### 設計理由
1. **型の一貫性**: `list | None` では、フロントエンド側で常に null チェックが必要になる
2. **空配列の意味**: 「割当なし」という状態を明示的に表現できる
3. **フロントエンドの簡略化**: 常に配列として扱えるため、`map()` などの配列メソッドが安全に使用できる

### 将来的な拡張
- 複数担任・学年主任・教科担当などの拡張を見据えた設計
- 現状は「1ユーザーにつき1つの `teacher_assignment` を主に扱う」前提
- 将来的には複数割当への対応を想定

## レスポンス構造の統一

### 2026/02/06 時点の設計
- `primary_assignment`: 一覧表示用の代表割当（単一オブジェクト）
- `teacher_assignments`: 詳細・編集用の全割当情報（配列）
- 両者を明確に分離することで、責務を明確化

## 責務分離の方針

### 認証処理（auth_service）
- トークン生成・検証
- ログイン/ログアウトの制御

### ユーザー情報集約（user_service）
- ロール別の情報取得
- 割当情報の整形・集約
- `build_admin_user_list` / `resolve_teacher_display_role` の実装を参考
