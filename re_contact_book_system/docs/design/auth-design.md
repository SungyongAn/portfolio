# 認証設計書（Auth Design）

本ドキュメントは、本システムにおける認証・認可設計の方針および、
その設計判断の背景を整理するものである。

本設計は CHANGELOG（2026/01/16, 01/23, 01/24）に記録された試行錯誤を整理・統合した結果である。

---

## 1. 設計方針

### 1.1 基本方針

- フロントエンドは **可能な限りステートレス** とする
- アクセストークンは短命とし、**メモリ上でのみ管理**する
- リフレッシュトークンは **HttpOnly Cookie** で管理し、
  フロントエンドからの直接アクセスを禁止する
- 認証エラーは **axios interceptor による一元ハンドリング**を行う

---

## 2. トークン構成

### 2.1 アクセストークン

- 保存先：フロントエンドのメモリ（Pinia）
- 役割：API リクエストの認証
- 有効期限：短命（分単位）

### 2.2 リフレッシュトークン

- 保存先：HttpOnly Cookie
- 役割：アクセストークン再発行
- 特徴：
  - JavaScript から参照不可
  - XSS 攻撃耐性を確保

---

## 3. トークン更新方式

### 3.1 採用方式：レスポンス起点方式

#### 処理概要

1. API レスポンスが 401 を返却
2. interceptor が検知
3. refresh API を呼び出し
4. 成功時：元リクエストを再実行
5. 失敗時：ログアウト処理

#### 採用理由

- タイマー方式では、**不要な更新・状態ズレ**が発生しやすい
- 実際の API 利用状況に即した更新が可能
- 実装・デバッグが単純になる

---

## 4. 再帰呼び出し防止

### 4.1 課題

- refresh API 自身が 401 を返すと、無限ループが発生する可能性がある

### 4.2 対応

- interceptor 内で **refresh API 呼び出し中フラグ**を管理
- 再帰的な refresh 呼び出しを抑止

---

## 5. 採用しなかった設計

### 5.1 トークン更新タイマー方式

#### 不採用理由

- 画面遷移・非アクティブ状態との整合が取りづらい
- 実装が複雑化しやすい
- 実挙動の検証コストが高い

## 7. モジュール構成と責務分離

### 7.1 構成方針

認証機能は責務ごとに明確に分離し、  
循環 import や起動時エラーを防ぐ構成とする。

- router / service / utils / dependencies を明確に分離
- __init__.py による再 export は行わない
- 下位レイヤーは上位レイヤーを import しない

### 7.2 各モジュールの責務

- routers/auth.py  
  HTTP リクエスト・レスポンス制御のみを担当

- services/auth_service.py  
  認証ユースケース（login / refresh）を担当

- utils/password_utils.py  
  パスワードのハッシュ化・検証を行う純粋関数群

- utils/token_utils.py  
  JWT の生成・検証などトークン関連処理を担当

- dependencies/auth.py  
  FastAPI Depends 用の認証依存関数を定義

### 7.3 import 設計と循環参照対策

- service から utils への import は許可
- utils から service への import は禁止
- router は service のみを参照
- __init__.py ではシンボルを再 export しない

これにより、認証機能の拡張時にも
循環 import や ImportError が発生しない構成としている。
